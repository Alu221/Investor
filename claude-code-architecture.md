## 2. Компоненты системы

### 2.1 Telegram → iia Bot

| Компонент | Описание |
|-----------|----------|
| Telegram Bot API | Получение/отправка сообщений |
| iia Bot (index.js) | Основной процесс бота на Node.js |
| Deepgram STT | Распознавание голосовых сообщений |
| Медиа-обработка | Скачивание фото/файлов во временные пути |
| Settings Manager | Управление env переменными, токенами |
| Cron Manager | Задачи по расписанию |
| Monitor | Фоновый мониторинг (конфиг: ~/.iia/monitor/config.json) |

### 2.2 iia Bot → Claude Code

Бот вызывает Claude Code одним из способов:

**Способ A — CLI spawn:**
```
spawn('claude', ['--print', '--session', sessionId], {
  env: {
    CLAUDE_CODE_OAUTH_TOKEN: '...',
    GITHUB_TOKEN: '...',
    VERCEL_TOKEN: '...',
    DEEPGRAM_API_KEY: '...',
    SERPER_API_KEY: '...',
    // все ключи из /settings
  },
  cwd: '/home/client/workspace'
})
```

**Способ B — Claude Agent SDK (Node.js):**
```
import { ClaudeAgent } from '@anthropic-ai/claude_agent_sdk';
const agent = new ClaudeAgent({
  oauthToken: process.env.CLAUDE_CODE_OAUTH_TOKEN,
  workingDirectory: '/home/client/workspace',
  // ...
});
const response = await agent.chat(userMessage);
```

## 3. Жизненный цикл сообщения

### Полный путь одного сообщения:

```
Шаг 1: Пользователь → Telegram
   "Покажи файл package.json"

Шаг 2: Telegram Bot API → iia Bot
   {
     "message": {
       "text": "Покажи файл package.json",
       "from": { "id": 12345 },
       "chat": { "id": 12345 }
     }
   }

Шаг 3: iia Bot обработка
   - Определяет тип (текст/голос/фото)
   - Загружает контекст сессии
   - Формирует промпт с системными инструкциями

Шаг 4: iia Bot → Claude Code
   - Передаёт текст + контекст
   - env с токенами
   - cwd = ~/workspace

Шаг 5: Claude Code → Anthropic API (запрос #1)
   messages: [{ role: "user", content: "Покажи файл package.json" }]
   tools: [Read, Write, Edit, Bash, Glob, Grep, Agent, ...]

Шаг 6: Anthropic API → Claude Code (ответ #1)
   {
     "role": "assistant",
     "content": [{
       "type": "tool_use",
       "id": "toolu_01ABC",
       "name": "Read",
       "input": { "file_path": "/home/client/workspace/package.json" }
     }]
   }

Шаг 7: Claude Code исполняет tool
   - Permission check: Read разрешён? → Да
   - fs.readFile("/home/client/workspace/package.json")
   - Результат: содержимое файла

Шаг 8: Claude Code → Anthropic API (запрос #2)
   messages: [
     { role: "user", content: "Покажи файл package.json" },
     { role: "assistant", content: [{ type: "tool_use", name: "Read", ... }] },
     { role: "user", content: [{ type: "tool_result", content: "<содержимое файла>" }] }
   ]

Шаг 9: Anthropic API → Claude Code (ответ #2)
   {
     "role": "assistant",
     "content": [{
       "type": "text",
       "text": "Вот содержимое package.json:\n..."
     }]
   }

Шаг 10: Claude Code → iia Bot
   Финальный текст ответа

Шаг 11: iia Bot → Telegram
   sendMessage({ chat_id: 12345, text: "...", parse_mode: "HTML" })

Шаг 12: Telegram → Пользователь
   Отображение в чате
```

### Мульти-tool цикл:

```
Если модель вызывает 3 инструмента подряд:

API запрос #1 → tool_use (Read)
  ↓ исполнение
API запрос #2 → tool_use (Edit)
  ↓ исполнение
API запрос #3 → tool_use (Read) — проверка результата
  ↓ исполнение
API запрос #4 → text (финальный ответ)

= 4 запроса к API на одно сообщение пользователя
```

### Параллельные tool вызовы:

```
API запрос #1 → [tool_use (Read file1), tool_use (Read file2), tool_use (Bash)]
  ↓ исполняются параллельно
API запрос #2 → [tool_result #1, tool_result #2, tool_result #3]
  ↓
API запрос #3 → text (финальный ответ)

= 2 запроса к API, но 3 инструмента выполнены
```

## 4. Модель (LLM)

### Текущая модель
- **Название:** Claude Opus 4.6
- **Model ID:** claude-opus-4-6
- **Контекст:** ~200K токенов
- **Knowledge cutoff:** Май 2025

### Что модель умеет
- Генерировать текст (ответы)
- Генерировать tool_use (вызовы инструментов)
- Анализировать изображения (мультимодальность)
- Работать с длинным контекстом
- Рассуждать о коде, архитектуре, дебаге

### Что модель НЕ умеет (делает Runtime)
- Выполнять код
- Читать файлы напрямую
- Делать HTTP запросы
- Запускать процессы
- Запоминать между сессиями (без внешней памяти)

### Доступные модели для подагентов
| Model ID | Описание | Когда использовать |
|----------|----------|-------------------|
| opus | Claude Opus 4.6 | Сложные задачи, архитектура |
| sonnet | Claude Sonnet 4.6 | Баланс скорость/качество |
| haiku | Claude Haiku 4.5 | Быстрые простые задачи |

## 5. Tool Use Protocol

### Формат запроса модели (tool_use)

```json
{
  "type": "tool_use",
  "id": "toolu_01XYZ123",
  "name": "Bash",
  "input": {
    "command": "git status",
    "description": "Show working tree status"
  }
}
```

### Формат результата (tool_result)

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01XYZ123",
  "content": "On branch main\nnothing to commit, working tree clean"
}
```

### Формат ошибки

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01XYZ123",
  "is_error": true,
  "content": "Permission denied: /etc/shadow"
}
```

### Правила протокола
1. Модель генерирует tool_use → Runtime исполняет → модель получает tool_result
2. Модель может генерировать несколько tool_use в одном ответе (параллельные)
3. Цикл продолжается пока модель не сгенерирует текст вместо tool_use
4. Каждый tool_use имеет уникальный id для связки с tool_result
5. Runtime может отклонить вызов (permission denied) — модель увидит ошибку

### Поток данных

```
Модель                    Runtime                   Файловая система
  │                          │                           │
  ├─── tool_use(Read) ──────►│                           │
  │                          ├─── fs.readFile() ────────►│
  │                          │◄── содержимое ────────────┤
  │◄── tool_result ──────────┤                           │
  │                          │                           │
  ├─── tool_use(Edit) ──────►│                           │
  │                          ├─── string replace ───────►│
  │                          │◄── ok ───────────────────┤
  │◄── tool_result ──────────┤                           │
  │                          │                           │
  ├─── text("Готово") ──────►│                           │
  │                          ├─── → iia bot → Telegram   │
```

## 6. Все инструменты (Tools)

### 6.1 Read — Чтение файлов

```
Реализация: Node.js fs.readFile()
Вход: file_path, [offset], [limit], [pages]
Выход: содержимое с номерами строк (cat -n формат)

Лимиты:
- Макс 2000 строк по умолчанию
- Строки >2000 символов обрезаются
- PDF: макс 20 страниц за запрос

Поддерживаемые форматы:
- Текстовые файлы (любые)
- Изображения (PNG, JPG, GIF, WebP) → визуально
- PDF → текст, с указанием страниц
- Jupyter Notebooks (.ipynb) → ячейки + вывод
- Бинарные → ошибка

Примеры:
  Read("/home/client/workspace/index.js")
  Read("/home/client/workspace/doc.pdf", pages="1-5")
  Read("/tmp/iia_photo.jpg")  → модель видит изображение
```

### 6.2 Write — Запись файлов

```
Реализация: Node.js fs.writeFile()
Вход: file_path, content
Выход: подтверждение

Правила:
- Полная перезапись файла
- Если файл существует → ОБЯЗАН сначала прочитать через Read
- Для частичных правок → использовать Edit
- Создаёт директории если нет? → Нет, директория должна существовать
```

### 6.3 Edit — Точечная правка

```
Реализация: Поиск exact string в файле → замена
Вход: file_path, old_string, new_string, [replace_all]
Выход: подтверждение

Алгоритм:
1. Прочитать файл
2. Найти old_string (exact match)
3. Если не найден → ошибка
4. Если найден несколько раз и replace_all=false → ошибка
5. Заменить → записать файл

Ограничения:
- old_string должен быть уникален (или replace_all=true)
- Индентация должна совпадать точно
- Модель не должна включать номера строк из Read
```

### 6.4 Glob — Поиск файлов

```
Реализация: fast-glob (npm пакет)
Вход: pattern, [path]
Выход: список файлов, отсортированных по дате модификации

Паттерны:
  "**/*.js"         → все JS файлы рекурсивно
  "src/**/*.ts"     → TS файлы в src/
  "*.{json,yaml}"   → JSON и YAML в текущей директории
  "!node_modules"   → исключить node_modules
```

### 6.5 Grep — Поиск по содержимому

```
Реализация: ripgrep (rg) — бинарный инструмент
Вход: pattern (regex), [path], [output_mode], [glob], [type], [контекст], [multiline]
Выход: зависит от output_mode

output_mode:
  "files_with_matches" (default) → список путей файлов
  "content"                      → совпадающие строки с контекстом
  "count"                        → количество совпадений

Примеры:
  Grep("function handleClick", type="js")
  Grep("TODO|FIXME", output_mode="content", -C=2)
  Grep("import.*React", glob="*.tsx")
```

### 6.6 Bash — Выполнение команд

```
Реализация: child_process.spawn('/bin/bash', ['-c', command])
Вход: command, description, [timeout], [run_in_background]
Выход: stdout + stderr + exit code

Среда:
- Shell: bash
- Рабочая директория: сохраняется между вызовами (через cwd tracking)
- Состояние shell: НЕ сохраняется (переменные, alias, etc.)
- env: наследуется от процесса Claude Code (включая все токены)
- Таймаут: 120с по умолчанию, макс 600с
- Песочница: ограничивает опасные операции

run_in_background:
- Команда запускается, управление возвращается сразу
- Уведомление приходит когда завершится
- Для долгих процессов (npm install, build, etc.)
```

### 6.7 Agent — Подагенты

```
Реализация: Запуск нового экземпляра Claude Code как subprocess
Вход: prompt, description, [subagent_type], [model], [run_in_background], [isolation], [resume]
Выход: текстовый результат работы агента

Типы подагентов:
┌──────────────────┬────────────────────────────────┬──────────────────────┐
│ Тип              │ Инструменты                    │ Когда использовать   │
├──────────────────┼────────────────────────────────┼──────────────────────┤
│ general-purpose  │ ВСЕ инструменты                │ Сложные задачи       │
│ Explore          │ Read, Glob, Grep, Bash, Web    │ Исследование кода    │
│                  │ (НЕТ: Agent, Edit, Write)      │                      │
│ Plan             │ Read, Glob, Grep, Bash, Web    │ Архитектурные планы  │
│                  │ (НЕТ: Agent, Edit, Write)      │                      │
│ statusline-setup │ Read, Edit                     │ Настройка UI         │
└──────────────────┴────────────────────────────────┴──────────────────────┘

isolation: "worktree"
- Создаёт временный git worktree
- Агент работает в изолированной копии репо
- Если нет изменений → worktree удаляется
- Если есть → возвращает путь и ветку

resume:
- Передаёт agent_id предыдущего вызова
- Агент продолжает с полным контекстом предыдущей работы

Особенности:
- У каждого подагента СВОЁ контекстное окно
- Результат НЕ виден пользователю — основной агент должен пересказать
- Несколько агентов могут работать параллельно
- Фоновый агент уведомляет когда закончит
```

### 6.8 WebSearch — Поиск в интернете

```
Реализация: Anthropic proxy или Serper API ($SERPER_API_KEY)
Вход: поисковый запрос
Выход: результаты поиска (заголовки, сниппеты, URL)

Статус: Deferred tool (загружается через ToolSearch)
```

### 6.9 WebFetch — Загрузка страниц

```
Реализация: HTTP GET → парсинг HTML → извлечение текста
Вход: URL
Выход: текстовое содержимое страницы

Статус: Deferred tool
```

### 6.10 TodoWrite — Трекинг задач

```
Реализация: Внутренний список задач в контексте разговора
Вход: список задач с id, описанием, статусом
Статусы: pending, in_progress, completed

Особенности:
- Видно только модели (для самоорганизации)
- Не сохраняется между сессиями
- Помогает не потерять прогресс в сложных задачах
```

### 6.11 Skill — Вызов навыков

```
Реализация: Загрузка SKILL.md промпта → выполнение инструкций
Вход: skill (имя), [args]

Доступные навыки:
- keybindings-help → настройка горячих клавиш
- simplify → ревью кода на качество
- loop → повторяющийся запуск
- claude-api → помощь с Anthropic SDK

Источники навыков:
- Встроенные в Claude Code
- ~/.claude/skills/<name>/SKILL.md — пользовательские
- ~/workspace/.claude/skills/<name>/SKILL.md — проектные
```

### 6.12 CronCreate / CronList / CronDelete

```
Реализация: iia Bot регистрирует задачу в своём планировщике
Вход (create): расписание + промпт
Вход (list): нет параметров
Вход (delete): id задачи

Как работает:
1. Модель вызывает CronCreate с расписанием и промптом
2. iia Bot сохраняет в свою базу
3. По расписанию бот запускает Claude Code с указанным промптом
4. Результат отправляется пользователю в Telegram
```

### 6.13 ToolSearch — Загрузка отложенных инструментов

```
Реализация: Поиск по списку deferred tools → загрузка JSON Schema
Вход: query, [max_results]

Зачем: Не все инструменты загружены в system prompt (экономия токенов).
Deferred tools известны только по имени. ToolSearch загружает полную схему,
после чего инструмент можно вызвать.
```

## 12. Deferred Tools

### Концепция

```
Проблема: 18+ инструментов × JSON Schema = много токенов в system prompt
Решение: Загружать схемы инструментов по требованию

При старте сессии:
- Основные tools (Read, Write, Edit, Bash, Glob, Grep, Agent, Skill, ToolSearch)
  → полные JSON Schema загружены
- Остальные (WebSearch, WebFetch, Todo, Cron, Plan, Worktree, Notebook, etc.)
  → известны только по имени

Когда нужен deferred tool:
1. Модель вызывает ToolSearch("select:WebSearch")
2. Runtime загружает полную JSON Schema
3. Теперь модель может вызвать WebSearch с правильными параметрами
```

### OAuth vs API Key

```
API Key (sk-ant-...):
├── Создаётся на console.anthropic.com
├── Оплата за токены ($$/1M tokens)
├── Прямой REST: POST /v1/messages
├── Без инструментов (только чат)
└── Ты сам управляешь всем

OAuth Token (CLAUDE_CODE_OAUTH_TOKEN):
├── Получается через OAuth flow (логин в claude.ai)
├── Привязан к подписке (фикс $/месяц)
├── Используется Claude Code Runtime
├── Все инструменты (Read, Bash, etc.)
├── Autocompact, sessions, agents
└── Лимиты = лимиты подписки
```

## 18. Ограничения

### Технические
- Контекст ~200K токенов (после — autocompact, потеря деталей)
- Bash таймаут: макс 10 минут на команду
- Read: макс 2000 строк, строки >2000 символов обрезаются
- PDF: макс 20 страниц за раз
- Нет interactive shell (нет vim, nano, git -i, ssh interactive)
- Нет GUI (нет браузер, X11, desktop приложения)
- Нет daemon mode (процессы умирают с сессией)

### Безопасности
- Не видит ~/.iia/.env, state.json, sessions/
- Не может kill/pkill/sudo
- Не может модифицировать код бота
- Sandbox ограничивает опасные Bash команды
- Не может показывать API ключи в чате

### Модели
- Knowledge cutoff: Май 2025
